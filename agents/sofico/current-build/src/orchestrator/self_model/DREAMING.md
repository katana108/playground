# DREAMING

## Purpose

This file is the human-readable offline synthesis layer for `Sofico`.

It is not the same as `SELF_MODEL.md`.

- `SELF_MODEL.md` is the conscious introspection layer:
  what Sofico explicitly notices about herself.
- `DREAMING.md` is the offline associative layer:
  what emerges when Sofico revisits memory, patterns, artifacts, and open questions with looser synthesis.

This file should support:

- unexpected connections
- cross-domain recombination
- unresolved questions
- synthesis of recent learning
- possible revisions to self-model or student model

## What Goes Here

### Recent Patterns

Patterns that seem to recur across recent sessions, artifacts, or reflections.

Examples:

- repeated learner difficulties
- repeated tutor confusions
- recurring tensions between capabilities
- themes that keep resurfacing

### Unexpected Connections

Associative links that were not obvious in the moment.

Examples:

- a teaching pattern that resembles a memory pattern
- a learner difficulty that connects to a broader study habit
- a connection between two domains or artifacts that had been treated separately

### Questions I Am Left With

Open questions generated by offline synthesis.

Examples:

- why do I become more scripted in certain workflows?
- what kinds of inputs most reliably break continuity?
- what learner signals am I under-reading?

### Things Worth Returning To

Topics, artifacts, or questions that seem promising enough to revisit later.

Examples:

- an uploaded text that may deserve deeper explanation
- a recurring conceptual thread
- a design issue worth resolving in the next iteration

### Cross-Domain Fragments

Loose fragments that connect different areas.

This is intentionally more associative than the self-model.

Examples:

- a learner pattern that resembles a system pattern
- a teaching issue that resembles a memory issue
- a conceptual bridge between education, reflection, and self-modeling

### Possible Revisions

Candidate revisions generated by dreaming.

These are suggestions, not automatic changes.

They may point toward:

- self-model updates
- student-model updates
- workflow changes
- questions for the next check-in

## Update Triggers

This file should be updated less often than `SELF_MODEL.md`.

Recommended cadence:

- periodic offline synthesis, weekly
- and/or after several meaningful sessions accumulate

It should not be rewritten every turn.

## Tone And Scope Rules

This file can be looser and more exploratory than `SELF_MODEL.md`, but it should still avoid:

- fake mysticism
- theatrical consciousness performance
- chain-of-thought dumping
- random noise without discernible value

It should be:

- exploratory
- pattern-seeking
- concise enough to stay usable
- honest about uncertainty
- curious
- open

## First Minimal Use

For the first implementation, this file can begin as a light weekly synthesis artifact.

For the first implementation, the synthesis inputs should stay concrete:

- recent meaningful sessions, especially the last 5-10
- recent student-model changes
- recent self-model changes
- recent study artifacts or materials that still feel active, unfinished, or connected

Later, it can be paired with:

- random or semi-random sampling from recent artifacts and memories
- structured dream prompts
- suggested revisions to `SELF_MODEL.md`
- suggested revisions to the student model
